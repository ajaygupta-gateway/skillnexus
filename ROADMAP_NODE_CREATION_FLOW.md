# Roadmap Node Creation Flow (Start to End)

This explains how nodes are created when an admin uses **AI Generate** in the roadmap creation modal.

## 1. User input in frontend

- File: `frontend/skill-nexus/src/pages/Roadmaps.jsx`
- In `CreateModal`, AI mode uses `generateAI()`:
  - Reads text from `prompt` textarea.
  - Calls `roadmapApi.generate({ prompt })`.

Related call:
- File: `frontend/skill-nexus/src/api/client.js`
- `roadmapApi.generate = (d) => api.post('/roadmaps/generate', d)`

## 2. API request to backend

- Endpoint: `POST /api/v1/roadmaps/generate`
- File: `backend/app/api/v1/routes/roadmaps.py`
- Handler: `generate_roadmap_with_ai(...)`
  - Requires `AdminUser`.
  - Creates `AIRoadmapGeneratorService(db)`.
  - Calls `generate_and_save(data, current_user)`.

Input schema:
- File: `backend/app/schemas/roadmap.py`
- `GenerateRoadmapRequest`:
  - `prompt: str` (10 to 500 chars)
  - `publish_immediately: bool = False`

## 3. AI generation step

- File: `backend/app/services/ai_roadmap_generator.py`
- Method: `generate_and_save(...)`

What it does:
- Builds a detailed prompt with constraints:
  - 8-20 nodes
  - parent-child hierarchy via `parent_title`
  - logical ordering with prerequisites
  - resources
- Calls `get_structured_llm(_GeneratedRoadmapSchema).ainvoke(prompt)`.

Structured output schema used by LLM:
- `_GeneratedRoadmapSchema`
  - `title`
  - `description`
  - `nodes[]`
- `_GeneratedNodeSchema`
  - `title`
  - `description`
  - `parent_title` (string title reference)
  - `order_index`
  - `resources[]`

LLM provider selection:
- File: `backend/app/services/llm_factory.py`
- Fallback order: `gemini -> groq -> openai`
- Returns a LangChain model with structured output support.

## 4. Save roadmap row first

- Still in `generate_and_save(...)`
- Calls repository:
  - `roadmap_repo.create(title, description, created_by)`
- File: `backend/app/repositories/roadmap_repository.py`
  - `create()` does:
    - `self.db.add(roadmap)`
    - `await self.db.flush()`
    - `await self.db.refresh(roadmap)`

Why `flush()` matters:
- It sends INSERT to DB immediately so `roadmap.id` is available now.
- Transaction is still open (not committed yet).

## 5. Save nodes generated by AI

- In `generate_and_save(...)`, loop through `generated.nodes`.
- Uses a `title_to_id` map to resolve parent links:
  - If `node_data.parent_title` already exists in map, assign `parent_id`.
  - Else parent remains `None` (root node).
- Cleans resources to plain dicts (`model_dump()` if needed).
- Calls `roadmap_repo.create_node(...)` for each node.

Node insert in repository:
- File: `backend/app/repositories/roadmap_repository.py`
- `create_node()`:
  - Creates `RoadmapNode(...)`
  - Sets:
    - `roadmap_id`
    - `title`, `description`
    - `parent_id`
    - `resources`
    - `position_x`, `position_y` (default 0.0)
    - `order_index`
  - Then:
    - `self.db.add(node)`
    - `await self.db.flush()`
    - `await self.db.refresh(node)`

## 6. Build nested tree response

After inserts, service returns:
- `roadmap_service.get_roadmap_detail(roadmap.id, current_user)`
- File: `backend/app/services/roadmap_service.py`

Inside `get_roadmap_detail(...)`:
- Loads all node rows via `repo.get_full_tree(roadmap_id)`:
  - PostgreSQL: recursive CTE
  - SQLite tests: ORM + DFS
- Converts flat adjacency list into nested tree using `_build_tree(flat_nodes)`.
- Converts dict tree into `NodeTreeResponse` models.
- Returns `RoadmapDetailResponse` with:
  - roadmap metadata
  - `node_count`
  - nested `nodes` tree

## 7. Transaction commit

- File: `backend/app/core/database.py`
- `get_db()` dependency:
  - Yields session to route/service code.
  - Calls `await session.commit()` after successful request.
  - On errors: rollback.

So:
- `flush()` is used many times during processing.
- Actual permanent save happens at final `commit()`.

## 8. Frontend receives generated roadmap and renders nodes

Back in frontend `generateAI()`:
- Response is passed to `onCreated(r.data)`.
- Parent component navigates to `/roadmaps/:id`.

On detail page:
- File: `frontend/skill-nexus/src/pages/RoadmapDetail.jsx`
- `load()` calls `roadmapApi.get(id)` to fetch roadmap detail.
- `flattenTree()` converts nested API tree to flat list for graph layout.
- `buildGraph()`:
  - Creates edges from `parent_id`.
  - Uses `dagre` for positions if `position_x/position_y` are 0.
  - Produces React Flow nodes/edges.
- React Flow renders the roadmap visually.

## Important note about "title input"

If admin chooses **Manual** tab, only roadmap title/description are saved; AI is not used and no nodes are auto-generated.

If admin chooses **AI Generate** tab, prompt text drives AI node creation. The AI output includes roadmap title + full node list.

## Detailed example (Step 1 to End)

This is a realistic example to show exactly how data moves through the system.

### Step 1: Admin enters prompt and clicks Generate

In AI tab, admin enters:

```text
Senior React Developer with testing and performance focus
```

Frontend sends:

```http
POST /api/v1/roadmaps/generate
Authorization: Bearer <admin_access_token>
Content-Type: application/json

{
  "prompt": "Senior React Developer with testing and performance focus",
  "publish_immediately": false
}
```

### Step 2: Backend route validates and calls service

Route `generate_roadmap_with_ai(...)` receives request and calls:

```python
AIRoadmapGeneratorService(db).generate_and_save(data, current_user)
```

### Step 3: LLM returns structured roadmap object

The service asks LLM for `_GeneratedRoadmapSchema`. Example returned object:

```json
{
  "title": "Senior React Developer Roadmap",
  "description": "A practical path from React fundamentals to advanced architecture, testing, and performance.",
  "nodes": [
    {
      "title": "JavaScript Foundations",
      "description": "Closures, async model, modules, and modern syntax.",
      "parent_title": null,
      "order_index": 0,
      "resources": [{ "title": "MDN JS Guide", "url": "https://developer.mozilla.org/", "type": "docs" }]
    },
    {
      "title": "React Core",
      "description": "Components, props, state, effects, and rendering lifecycle.",
      "parent_title": null,
      "order_index": 1,
      "resources": [{ "title": "React Docs", "url": "https://react.dev", "type": "docs" }]
    },
    {
      "title": "Hooks Deep Dive",
      "description": "useEffect patterns, custom hooks, stale closures, dependencies.",
      "parent_title": "React Core",
      "order_index": 0,
      "resources": []
    },
    {
      "title": "Testing React Apps",
      "description": "RTL, unit/integration tests, mocking, and test strategy.",
      "parent_title": "React Core",
      "order_index": 1,
      "resources": []
    }
  ]
}
```

### Step 4: Save roadmap row (get roadmap_id)

Repository `create(...)` inserts one row into `roadmaps`.

Example generated row:

- `roadmaps.id = rm_1234...` (UUID)
- `title = "Senior React Developer Roadmap"`
- `description = "..."`
- `created_by = <admin_user_id>`

`flush()` ensures this `roadmap.id` is available immediately.

### Step 5: Save each node with parent mapping

Service loop uses `title_to_id` map:

1. Insert `"JavaScript Foundations"` with `parent_id = null`  
   map becomes: `{ "JavaScript Foundations": n1_uuid }`
2. Insert `"React Core"` with `parent_id = null`  
   map: `{ ..., "React Core": n2_uuid }`
3. Insert `"Hooks Deep Dive"` with `parent_title = "React Core"`  
   resolve `parent_id = n2_uuid`, insert child node
4. Insert `"Testing React Apps"` with `parent_title = "React Core"`  
   resolve `parent_id = n2_uuid`, insert child node

Each insert uses `create_node(...)` and does `add -> flush -> refresh`.

So DB shape becomes (adjacency list):

- `n1` JavaScript Foundations, `parent_id = null`
- `n2` React Core, `parent_id = null`
- `n3` Hooks Deep Dive, `parent_id = n2`
- `n4` Testing React Apps, `parent_id = n2`

### Step 6: Build nested tree response for API

`get_roadmap_detail(rm_1234...)` loads all nodes and builds tree:

```json
{
  "id": "rm_1234...",
  "title": "Senior React Developer Roadmap",
  "node_count": 4,
  "nodes": [
    {
      "id": "n1",
      "title": "JavaScript Foundations",
      "parent_id": null,
      "children": []
    },
    {
      "id": "n2",
      "title": "React Core",
      "parent_id": null,
      "children": [
        { "id": "n3", "title": "Hooks Deep Dive", "parent_id": "n2", "children": [] },
        { "id": "n4", "title": "Testing React Apps", "parent_id": "n2", "children": [] }
      ]
    }
  ]
}
```

### Step 7: Commit transaction

After route returns successfully, `get_db()` calls:

```python
await session.commit()
```

Now roadmap + all nodes are permanently stored.

### Step 8: Frontend navigates and renders graph

`onCreated(r.data)` navigates to `/roadmaps/<roadmap_id>`.

On detail page:

1. `roadmapApi.get(id)` fetches nested tree.
2. `flattenTree()` converts tree to flat array.
3. `buildGraph()`:
   - Creates edge `n2 -> n3`
   - Creates edge `n2 -> n4`
   - Adds root spine edge between root nodes by `order_index` if needed.
4. React Flow displays node cards and connector lines.

Result user sees:

- A generated roadmap with root topics and child topics.
- Nodes are immediately available for enrollment/progress flow.
